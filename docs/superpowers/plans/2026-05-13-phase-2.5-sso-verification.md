# Phase 2.5: SSO Verification Helper Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build `lib/ghl-sso.ts` — a single pure function `verifySso(token)` that takes a GHL Marketplace SSO JWT, verifies its HS256 signature with `GHL_SSO_KEY`, and returns the decoded payload (or `null` on any failure). This is the only authoritative identity check in the app — every API route from Phase 4+ will route through it before any database access or workflow call.

**Architecture:** Pure function. No DB, no HTTP, no Next.js — just `jsonwebtoken.verify()` wrapped in a try/catch that fails closed. The token format is documented in spec §4 (HS256 JWT, signed by GHL with the app's SSO key from the Marketplace settings; payload contains `userId`, `companyId`, `locationId`, `role`). All decoding happens server-side; the encrypted JWT is the only thing that touches the network from the browser.

**Tech Stack:** `jsonwebtoken@^9` (runtime) + `@types/jsonwebtoken@^9` (dev). Vitest unit tests, no MSW, no PGlite — pure-function tests sign their own tokens with a known secret and assert how `verifySso` handles each shape.

**Spec reference:**
- §4 (postMessage + SSO flow) — wire-level shape, mentions `jsonwebtoken` + HS256
- §8 (`lib/` structure) — function signature: `verifySso(token: string): SsoPayload | null`. Fails closed — never throws.
- §14 item 1 (resolved decisions) — confirmed HS256 JWT with `GHL_SSO_KEY` shared secret

**Out of scope for Phase 2.5:**
- Any route handler that *uses* `verifySso` — that's Phase 4 (buttons CRUD with `withAdminSso` HOF) and Phase 5 (enroll with `withSso` HOF).
- The `withSso` / `withAdminSso` higher-order route wrappers — those live in `lib/auth.ts` (Phase 4 introduces them).
- The client-side iframe code that asks GHL for the JWT via `postMessage` — Phase 6 / Phase 7 widget + admin client components.
- Refresh / expiry handling — JWT `exp` claim is checked by `jsonwebtoken.verify()`; we don't need a separate refresh flow because GHL hands us a fresh JWT on every iframe load.
- Encrypted-at-rest storage of `GHL_SSO_KEY` — it's a server-only env var; same security posture as `GHL_CLIENT_SECRET`.

**Reference docs to skim before starting:**
- `jsonwebtoken` README on npm — focus on `jwt.verify(token, secret, { algorithms: ['HS256'] })` and the error types it throws
- Spec §4 (linked above) — what fields are guaranteed in the payload
- The existing Phase 2 unit-test pattern in `lib/ghl-oauth.test.ts` — same shape of file applies here

**Important repo conventions (carry-overs):**
- Strict TypeScript; ESM throughout; Node 20.11+.
- Vitest with `globals: false` — explicit imports of `describe, expect, it`, etc. from `'vitest'`.
- MSW is wired in `vitest.setup.ts` with `onUnhandledRequest: 'error'`. This phase's tests make no HTTP calls so MSW is dormant.
- Tests live next to source: `lib/ghl-sso.ts` → `lib/ghl-sso.test.ts`.
- Env vars are read at point-of-use via `getEnv()` from `lib/env.ts` (Phase 2 Task 1). Tests stub via `vi.stubEnv` + `vi.unstubAllEnvs` in `afterEach`.

## How tests sign JWTs locally

Phase 2.5 is a pure-function phase: tests sign their own JWTs with `jsonwebtoken.sign()` using a known test secret, then call `verifySso` to assert behavior. Production never imports `jsonwebtoken.sign()` — only `verify()`. Importing both in tests is fine; the unused `sign` path doesn't ship to Vercel.

The test pattern looks like:

```ts
import jwt from 'jsonwebtoken'
const TEST_SECRET = 'test-sso-key'
const goodToken = jwt.sign(
  { userId: 'u1', companyId: 'c1', locationId: 'l1', role: 'admin' },
  TEST_SECRET,
  { algorithm: 'HS256', expiresIn: '1h' },
)
```

- [ ] **Step 4: Implement `lib/ghl-sso.ts`**

Create `lib/ghl-sso.ts`:

```ts
import jwt from 'jsonwebtoken'
import { getEnv } from './env'

export interface SsoPayload {
  userId: string
  companyId: string
  locationId: string
  role: string
}

/**
 * Verify a GHL Marketplace SSO JWT (HS256, signed by GHL with our app's
 * shared `GHL_SSO_KEY` from the Marketplace settings).
 *
 * Returns the narrow `SsoPayload` on success, or `null` on any failure
 * (bad signature, expired, wrong algorithm, malformed, missing required
 * fields). NEVER throws — every route handler treats `null` as 401,
 * so failing closed is the only safe behavior.
 *
 * Only the four documented payload fields are returned; GHL may send
 * additional fields (email, etc.) but consumers should ignore them
 * until the spec promises them.
 */
export function verifySso(token: string): SsoPayload | null {
  if (!token) return null

  const secret = getEnv('GHL_SSO_KEY')

  let decoded: unknown
  try {
    decoded = jwt.verify(token, secret, { algorithms: ['HS256'] })
  } catch {
    return null
  }

  if (typeof decoded !== 'object' || decoded === null) return null

  const payload = decoded as Record<string, unknown>
  if (
    typeof payload.userId !== 'string' ||
    typeof payload.companyId !== 'string' ||
    typeof payload.locationId !== 'string' ||
    typeof payload.role !== 'string'
  ) {
    return null
  }

  return {
    userId: payload.userId,
    companyId: payload.companyId,
    locationId: payload.locationId,
    role: payload.role,
  }
}
```

```ts
  it('returns null for an empty string', () => {
    expect(verifySso('')).toBeNull()
  })

  it('returns null for a malformed token (not a JWT)', () => {
    expect(verifySso('not.a.jwt')).toBeNull()
    expect(verifySso('justgarbage')).toBeNull()
  })

  it('returns null when the signature is invalid (signed with the wrong secret)', () => {
    const token = makeToken(
      { userId: 'u1', companyId: 'c1', locationId: 'l1', role: 'admin' },
      { algorithm: 'HS256' },
      'a-different-secret',
    )
    expect(verifySso(token)).toBeNull()
  })

  it('returns null when the token is expired', () => {
    const token = makeToken(
      {
        userId: 'u1',
        companyId: 'c1',
        locationId: 'l1',
        role: 'admin',
        // expired 60s ago
        exp: Math.floor(Date.now() / 1000) - 60,
      },
      { algorithm: 'HS256' },
    )
    expect(verifySso(token)).toBeNull()
  })

  it('returns null when the token was signed with a different algorithm (HS512)', () => {
    const token = makeToken(
      { userId: 'u1', companyId: 'c1', locationId: 'l1', role: 'admin' },
      { algorithm: 'HS512' },
    )
    expect(verifySso(token)).toBeNull()
  })

  it('returns null when the payload is missing required fields', () => {
    // Valid signature, but no userId/companyId/locationId/role
    const token = makeToken({ somethingElse: 'foo' })
    expect(verifySso(token)).toBeNull()
  })

  it('drops payload fields it does not promise (returns only the four documented fields)', () => {
    const token = makeToken({
      userId: 'u1',
      companyId: 'c1',
      locationId: 'l1',
      role: 'admin',
      email: 'tim@example.com',
      iat: 1700000000,
      exp: Math.floor(Date.now() / 1000) + 3600,
    })

    const result = verifySso(token)

    expect(result).toEqual({
      userId: 'u1',
      companyId: 'c1',
      locationId: 'l1',
      role: 'admin',
    })
    // The extras must not leak through
    expect(result).not.toHaveProperty('email')
    expect(result).not.toHaveProperty('iat')
    expect(result).not.toHaveProperty('exp')
  })
```

## What's NOT in Phase 2.5 (deferred)

- **`lib/auth.ts`** with `withSso(handler)` and `withAdminSso(handler)` higher-order route wrappers — Phase 4. They compose `verifySso` with the `X-GHL-SSO` request header parsing and the admin-role check, but they're route-handler glue, not part of the SSO primitive.
- **Reading the `X-GHL-SSO` header from `NextRequest`** — that's also `lib/auth.ts` in Phase 4.
- **The client-side `postMessage` flow** that requests the JWT from GHL — Phase 6 (widget) and Phase 7 (admin) client components.
- **Refresh handling** — every iframe load gets a fresh JWT from GHL, so we don't store or refresh them. The JWT `exp` claim is enforced by `jsonwebtoken.verify()`; an expired token returns `null`, the route returns 401, the client re-requests a fresh JWT via postMessage.
- **`role` value validation** beyond `typeof === 'string'`. Phase 4's admin gate will compare `decoded.role === 'admin'` — that comparison is the validation. We don't pre-enumerate roles here because GHL may add new ones.
- **An `iss` / `aud` / `jti` check.** GHL doesn't document setting these claims; if they appear in future tokens, we can add the claim checks via `jwt.verify(token, secret, { algorithms: ['HS256'], issuer, audience })` then.
